# TOEIC Master App (React Native / Expo)

完全オフラインで動作し、「個人の苦手」に徹底的にフォーカスしたTOEIC単語学習アプリです。
「わからない単語だけを効率よく叩き込む」ことを目的とし、ユーザーの記憶の定着をサポートする複数のテストモードと可視化機能を備えています。

## プロダクトのコア・コンセプト（開発の意図）
本アプリの最大のテーマは「苦手のあぶり出しと個別最適化」です。
最初からすべての単語を漫然と学ぶのではなく、「まだ覚えていない単語（★）」と「一度でも間違えた単語（⚠️）」を自動で仕分けし、それらを集中的に学習・テストできるサイクルを設計しました。

## こだわりの機能とUX（ユーザー体験）

### 1. 「学習」と「テスト」の完全分離
- 学習モード: 英語と日本語の意味を素早く確認し、音声読み上げでインプットに専念。
- テストモード: 記憶をアウトプットする実践的な4択テスト。

### 2. 徹底的な「苦手」トラッキング機能
- ★ 未学習・要復習マーク: 覚えていない単語には初期状態で「★」が付与され、「★克服テスト」で正解して初めて消去される達成感のある仕様。
- ⚠️ 過去ミス自動記録: テストで一度でも間違えた問題は自動的に「⚠️履歴」に登録され、苦手な単語だけを抽出した復習テストが可能。

### 3. 絶対に逃がさない「完走型テストアルゴリズム」
テスト中に間違えた問題はキューの最後に再追加され、「全問正解するまで何度でも出題され続ける」スパルタ仕様を採用。確実な記憶の定着を促します。

### 4. カスタマイズ可能な「My単語帳」
アプリに収録されていない単語をユーザー自身が追加（CRUD処理）できる機能を実装。追加した単語群に対しても、標準単語と同じく「学習・テスト」のサイクルを回すことができます。

### 5. モチベーションを維持する仕組み
- 初回起動時のオンボーディング（チュートリアル画面）によるスムーズな導入。
- 視覚的に進捗がわかる「プログレスバー」や「称号（ランク）システム」。
- いつでもやり直せる「学習データのリセット機能」。

## 技術的な課題解決
本プロジェクトは、生成AIをペアプログラミングのパートナーとして活用し超高速でプロトタイプ（MVP）を構築しました。しかし、「エンジニアとしてコードの品質に責任を持つ」ため、AIが生成したコードに対し以下のリファクタリングを自らの手で実施しています。

### 1. 配列シャッフルアルゴリズムの適正化
- 課題: 当初、AIはテスト問題のランダム出題に `Array.prototype.sort(() => 0.5 - Math.random())` を使用していましたが、これは確率的な偏り（バイアス）が生じる問題がありました。
- 解決策: 偏りをなくし、パフォーマンスも O(N) で済む Fisher-Yatesシャッフルアルゴリズム*を独自に実装し、テストの公平性を担保しました。

### 2. 例外処理（エラーハンドリング）の強化
- 課題: 初期コードでは `AsyncStorage` の読み書きエラーが `catch(e) {}` と握りつぶされており、データ破損時のリカバリーが不能でした。
- 解決策:非同期処理時のエラーログ出力を追加し、ユーザーに対してもアラート通知を行うよう適切にハンドリングしました。

## 今後の課題と展望（Next Steps）
現在は「動くものを最速でユーザーに届ける」というMVP開発のスピードを最優先したため、以下の技術的・環境的課題を残容しており、今後のアップデートで対応予定です。

### 1. コンポーネントの分割（アーキテクチャの改善）
現在はスピード優先のため、`index.tsx` という1つのファイルに複数の画面（Home, Test, Study等）が同居するモノリシックな構成になっています（約350行）。今後は保守性と可読性を高めるため、`src/screens/` や `src/components/` へのファイル分割（関心の分離）を行います。

### 2. ローカル開発環境からの脱却（スタンドアロン化）
現在は Expo Go を介してローカルサーバー（PCのコマンドプロンプト）を起動しておかないとアプリが動作しない状態です。今後は Expo EAS Build を用いて `.apk` (Android) や `.ipa` (iOS) のスタンドアロンアプリとしてビルドし、PCなしでも単体で動作・配布できる状態を目指します。
